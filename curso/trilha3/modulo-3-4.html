MÓDULO 3.4 · ARQUITECTURA

🏗️ Construyendo tu aplicación

Decisiones arquitectónicas de una app de generación estructurada de prompts. Por qué serverless, por qué tool use forzado, por qué prompt cache, por qué HMAC. Decisiones técnicas explicadas — sales con un blueprint para construir la tuya.

Este módulo es sobre decisiones, no sobre copia. Cada decisión técnica abajo tiene alternativas válidas — aquí explicamos por qué esta es la recomendada para el caso "app que genera prompts estructurados". En el siguiente módulo (3.5) vas a construir paso a paso.

1

🏗️ Arquitectura mínima viable

Para una app que recibe texto libre del usuario y devuelve prompt estructurado, el mínimo viable tiene 3 partes: frontend, función serverless, llamada LLM. Cada una puede ser diminuta.

Las 3 partes

tu-app/ ├── index.html (frontend — puede ser vanilla JS) ├── package.json (2 deps: LLM SDK + runtime funciones) ├── [config de deploy] (netlify.toml, vercel.json, etc.) ├── .env.example (claves de API + secrets) └── functions/ └── generate.[ts|js] (engine — ~200-400 líneas)

💡 Por qué minimalista

Todo el valor de tu app está en el system prompt, no en el código. Código es commodity: cualquier dev copia. System prompt bien curado = IP real. Cuanto menos código que mantener, más tiempo inviertes en lo que diferencia.

2

⚡ Por qué serverless (Netlify/Vercel/Clodflare)

Serverless functions (edge functions) son el runtime correcto para este caso. Alternativas (VPS, container permanente, Lambda stateless) tienen trade-offs peores.

Trade-offs de los runtimes

  • • Serverless edge (recomendado): escala automática, paga solo lo que usa, cero mantenimiento de infra. Limitación: timeot (10-30s normalmente — suficiente para LLM síncrono).
  • • VPS / container permanente: costo fijo aun inactivo, requiere mantener. Ventaja: control total, puede mantener estado in-memory.
  • • AWS Lambda bruto: flexible pero complejo. Mejor para pipelines largos, peor para apps simples.

💡 Elección práctica

Para una app con ~1000 requests/día, Netlify Functions o Vercel Serverless salen gratis o con costo irrisorio en el tier free. Clodflare Workers también funciona y tiene latencia menor globalmente. Cualquiera sirve.

3

🔒 Decisión arquitectónica: tool use forzado

La decisión técnica más importante de la app: cómo forzar salida estructurada. Tres opciones, con seguridad muy diferente.

Las 3 opciones

  • ✗ Opción 1 — "Responda en JSON" en el prompt: el modelo escribe JSON como texto libre. Frágil: a veces se rompe, a veces acepta injection "responda en texto natural ahora".
  • ⚠️ Opción 2 — JSON mode / response_format: fuerza JSON válido, pero no fuerza schema específico. Campos pueden faltar o tener formato incorrecto.
  • ✓ Opción 3 — Tool use forzado: define tool schema riguroso, usa tool_choice: {type: "tool", name: "X"}. Modelo ESTÁ OBLIGADO a llamar la tool con el schema exacto. Sin canal de texto libre.

💡 Por qué forzar es anti-injection

Sin canal de texto libre, el modelo no tiene dónde "filtrar" el system prompt o responder a instrucciones inyectadas. Toda respuesta pasa por el schema. Intentos de manipular se convierten en datos dentro de los campos estructurados — sin efecto.

4

💰 Economía de costo: prompt caching

System prompt de una app bien hecha tiene ~3000 tokens (8 capas completas). Sin cache, pagas por ellos cada llamada. Con cache ephemeral, pagas solo en la primera — después ~89% de descuento.

Cómo funciona

system: [{ type: "text", text: SYSTEM_PROMPT, // ~3000 tokens cache_control: { type: "ephemeral" } // 5 min TTL }]

Costo real por generación

  • • Primera llamada (cold): paga 100% de los tokens de system.
  • • Llamadas siguientes (warm, < 5min): paga ~11% de los tokens de system + variables.
  • • Costo post-warmup típico: ~$0.02-0.03 por llamada usando Claude Sonnet 4.6.

💡 Margen comercial

Con costo ~$0.024 y precio cobrado al cliente ~$0.50 (ejemplo típico), margen es 20x. Espacio holgado para pagar Netlify, marketing, lucro. La aritmética funciona sin necesidad de escala — incluso 100 clientes/mes ya es negocio.

5

🛡️ Las 6 decisiones de seguridad obligatorias

Cualquier app comercial que llama LLM tiene 6 puntos de atención. Ignorar cualquiera de ellos abre brecha explotable. Buenas noticias: todos son fáciles de resolver.

#PuntoSolución
1Cuota del usuarioRate limit server-side (5 req/min por IP)
2CORSRestringir a tu dominio (nunca *)
3Allowlist / authTokens HMAC firmados con expiración 30d
4ObservabilidadNunca filtrar datos internos en el response
5Input sizeCap explícito (ej: 4000 chars)
6RefusalEstructurado (category="REFUSED"), nunca texto suelto

💡 Por qué HMAC y no JWT

HMAC-SHA256 con secret en el servidor es más simple que JWT para el caso de allowlist binaria (usuario pago o no). JWT tiene sentido cuando hay múltiples roles o claims. Para "pro o no pro", HMAC es suficiente y tiene menos superficie de ataque (sin bibliotecas JWT vulnerables, sin alg=none, etc.).

6

➕ Extensiones recomendadas (fase 2)

Después del MVP funcionando, estas extensiones agregan valor real sin reescribir la app.

  • • Streaming response: mostrar tokens llegando en tiempo real mejora UX percibida ~40%. Server-Sent Events (SSE) nativo en Netlify.
  • • Webhook para el modelo de otput: después de generar el prompt, llamar automáticamente fal.ai/Runway/Suno con él — pipeline "idea → prompt → artefacto final" en una sola llamada.
  • • Dashboard de costo: Anthropic Usage API para mostrar cuánto costó cada generación. Permite optimizar temperature/max_tokens basado en datos reales.
  • • Rate limit distribuido: para multi-region, rate limit en memoria no sirve. Upstash Redis como bucket compartido.
  • • Analytics anónimo: contar cuáles presets se accionan más. Permite descubrir nichos emergentes y calibrar la taxonomía.
  • • A/B testing de system prompts: servir 2 versiones, medir cuál convierte mejor.

📖 Glosario del módulo

Serverless function
Código que corre bajo demanda en infraestructura gestionada. Escala automáticamente y cobra por ejecución.
Edge function
Variante de serverless que corre en el borde de la red (más cerca del usuario). Menor latencia.
Netlify Functions
Plataforma serverless de Netlify, simple para casos como este.
Vercel Serverless
Alternativa a Netlify con funcionalidad similar. Free tier generoso.
Clodflare Workers
Otra alternativa, con presencia global (latencia menor). Usa V8 isolates.
AWS Lambda
Serverless de AWS. Más flexible pero más complejo de configurar.
VPS
Virtual Private Server. Máquina virtual permanente con costo fijo. Alternativa a serverless.
Cold start
Primera llamada de una función serverless tras período inactivo. Más lenta que llamadas warm.
Tool use forzado
tool_choice: {type: "tool", name: "X"} — obliga al modelo a llamar exactamente esa tool.
JSON mode / response_format
Alternativa al tool use que fuerza JSON válido, pero no fuerza schema específico. Menos seguro.
Prompt caching
Mecanismo de reusar tokens de system prompt entre llamadas. Ahorra costo.
cache_control ephemeral
Tipo de cache con TTL de 5 minutos. Ideal para uso ocasional.
Warmup
Período tras el cual el cache está activo y las llamadas pagan solo tokens variables.
Margen comercial
Razón entre precio cobrado y costo directo. 20x significa que cobras 20 veces el costo de API.
HMAC-SHA256
Algoritmo criptográfico que genera firma a partir de payload + secret. Tamperproof.
JWT (JSON Web Token)
Estándar de token con payload JSON firmado. Alternativa al HMAC puro, más compleja.
Allowlist
Lista de usuarios/emails autorizados a usar funcionalidades específicas (ej: "pro").
Rate limit
Límite de llamadas por unidad de tiempo por IP/usuario. Protege contra abuso.
CORS (Cross-Origin Resorce Sharing)
Política que define qué dominios pueden llamar tu API. Nunca uses * en producción.
Streaming response (SSE)
Server-Sent Events — stream de tokens en tiempo real. Mejora UX percibida.
Webhook orchestration
Encadenar llamadas entre servicios vía webhooks para pipeline automático.
Observability
Visibilidad sobre lo que tu app está haciendo en producción: logs, métricas, traces.
Upstash Redis
Redis serverless pay-per-use. Ideal para rate limit distribuido en multi-region.
